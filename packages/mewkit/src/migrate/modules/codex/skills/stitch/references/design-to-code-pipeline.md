Orchestration workflows chaining Stitch design generation with implementation skills.

## Basic Pipeline

```
Text prompt → stitch-api-call.ts generate → stitch-api-call.ts export
           → stitch-write-output.ts → mk:frontend-design
```

### Steps

1. User describes desired UI in natural language
2. Run `stitch-api-call.ts generate` with the prompt
3. Show preview image (`imageUrl` from JSON output) to user for approval
4. Run `stitch-api-call.ts export <screen-id> --format all` to get CDN URLs
5. Pipe to `stitch-write-output.ts` to write `design.html`, `design.png`, `DESIGN.md`
6. Hand `DESIGN.md` path to `mk:frontend-design` for implementation

### Example

```bash
# 1. Generate
npx tsx .agents/skills/stitch/scripts/stitch-api-call.ts \
  generate "E-commerce product page with image gallery, price, reviews, add-to-cart"

# 2. Export + write
npx tsx .agents/skills/stitch/scripts/stitch-api-call.ts export <screen-id> --format all | \
  npx tsx .agents/skills/stitch/scripts/stitch-write-output.ts

# 3. Implement (mk:frontend-design reads DESIGN.md as design spec)
# Activate mk:frontend-design: "Implement from tasks/designs/<plan>/<screen-id>/DESIGN.md"
```

## Advanced Pipeline (with Variants)

```
Prompt → Generate → Variants → User Picks → Export + Write → Implement
```

### Steps

1. Generate base design
2. Generate 2-3 variants with different aspects (color, layout)
3. Show all variant image URLs to user
4. User selects preferred variant
5. Export and write selected variant
6. Hand off to implementation

### Example

```bash
# Generate with variants
npx tsx .agents/skills/stitch/scripts/stitch-api-call.ts \
  generate "SaaS pricing page" --variants 3

# User reviews images, picks variant 2 (screen-id from variants[] in JSON output)
npx tsx .agents/skills/stitch/scripts/stitch-api-call.ts export <variant-2-screen-id> --format all | \
  npx tsx .agents/skills/stitch/scripts/stitch-write-output.ts
```

## DESIGN.md Specification

`DESIGN.md` is an agent-readable markdown file containing extracted design tokens.
Generated by `stitch-write-output.ts` from the downloaded HTML.

### Structure

```markdown
# Design System

<!-- stitch-screen-id: <id> -->

## Layout
- `layout_type: top-nav-layout`

## Colors
- `bg-blue-600`
- `text-gray-900`

## Typography
- `text-2xl`
- `font-bold`

## Spacing
- `p-4`
- `gap-6`

## Components
- `<nav>`
- `<section>`

## Notes
- Generated from Stitch AI HTML export
- Tailwind CSS utility classes
- Pass to mk:frontend-design for implementation
```

### How Implementation Skills Consume It

1. Pass the DESIGN.md path to `mk:frontend-design`: "Implement from DESIGN.md at <path>"
2. `mk:frontend-design` reads `layout_type`, `components[]`, `colors[]` as constraints
3. DESIGN.md takes precedence over verbal design descriptions
4. If DESIGN.md is absent, skills fall back to text-based design

## Skill Routing

| Design Need | Skill | What It Does |
|-------------|-------|-------------|
| React/Vue/Svelte components | `mk:frontend-design` | Converts Tailwind HTML to framework components |
| Design token extraction | `mk:ui-design-system` | Extracts tokens from DESIGN.md for design systems |
| Full page with style guide | `mk:frontend-design` | Builds complete layouts with design spec |

## Handoff Protocol

1. `mk:stitch` writes artifacts to `tasks/designs/<plan-or-repo>/<screen-id>/`
2. Pass the DESIGN.md path to the implementation skill
3. Activate `mk:frontend-design`: "Implement from tasks/designs/…/DESIGN.md"
4. Implementation skill reads DESIGN.md, applies tokens, generates code
5. No modifications needed to existing skills — they read DESIGN.md opportunistically

## Quota-Aware Pipeline

Always check quota before generating:

```bash
# Pre-check
npx tsx .agents/skills/stitch/scripts/stitch-quota.ts check

# If exhausted (exit code 2):
# → Use mk:frontend-design for text-based design instead

# If credits available:
npx tsx .agents/skills/stitch/scripts/stitch-api-call.ts generate "<prompt>"
# (quota auto-tracked inside stitch-api-call.ts)
```
