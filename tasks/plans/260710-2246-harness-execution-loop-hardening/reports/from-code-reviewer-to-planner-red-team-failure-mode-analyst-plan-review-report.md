# Red-Team Failure-Mode Review — Harness Execution-Loop Hardening Plan

Reviewer role: FAILURE MODE ANALYST / FLOW TRACER. Scope: plan.md + 7 phase files vs current `meowkit` source. No linting/build run — plan-quality review backed by grep/read evidence.

## Finding 1: `active-run.json` has no concurrency control despite an in-repo locking precedent for the identical problem class

- **Severity:** Critical
- **Location:** phase-01-state-evidence-loop-core.md — Architecture diagram (lines 30-48), Implementation Steps 2 ("addBlocker/resolveBlocker, recordDecision... all through one atomic write helper"), Risk Assessment (lines 78-82)
- **Flaw:** The plan's only concurrency mitigation is "atomic writes (tmp+rename)" (line 28) and, in Risk Assessment, an explicit carve-out that ONLY applies to checkpoints: "Two checkpoint writers racing on richer payloads: keep last-writer-wins; payload stays pointer-only so a lost write costs freshness, not correctness" (line 82). `addBlocker`, `resolveBlocker`, and `recordDecision` are inherently read-modify-write operations on array fields (`blockers[]`, `open_decisions[]`) inside `active-run.json` — tmp+rename makes each individual write atomic at the filesystem level but does nothing to prevent two concurrent CLI invocations from both reading the same pre-mutation snapshot and one clobbering the other's array append (classic lost-update). Unlike the checkpoint file, `active-run.json` is the file Phase 2's Gate 1/pre-completion consume for blocking decisions — a lost blocker/decision is a correctness bug, not a freshness cost, and the plan's own language distinguishing "freshness" (checkpoint, acceptable) from correctness (active-run, unaddressed) is not applied to itself.
- **Failure scenario:** `mk:cook` (Build phase) calls `state add-blocker "flaky test X"` while a parallel PostToolUse hook or a second concurrent skill step calls `state set-phase 4` in the same window. Both processes read active-run.json before either writes; the later rename wins and silently drops the other's field-level change. The blocker vanishes from state; Phase 2's gate/resume logic reports a clean state that isn't, and the "kill/resume drill restores blockers" success criterion (Phase 1 line 74) can pass in testing (single-writer drill) while failing in the concurrent case the drill never exercises.
- **Evidence:** `meowkit/.claude/hooks/lib/checkpoint-utils.cjs:1-4` documents the existing last-writer-wins acceptance for checkpoints only; `meowkit/packages/mewkit/src/core/file-lock.ts:1-5` ("PID-based advisory file lock generalized over an arbitrary lock path... bounded spin") is an existing, reusable primitive already used by `immediate-capture-handler.cjs:11` ("Per-file lock: all writers targeting the same topic file acquire the same lock") and `migrate/reconcile/portable-registry-lock.ts:1`. Phase 1 does not reference `file-lock.ts` anywhere in its Related Code Files or Implementation Steps.
- **Suggested fix:** Reuse `core/file-lock.ts` around the read-modify-write section of every mutating `session-state/` writer function (`addBlocker`, `resolveBlocker`, `recordDecision`, `setPhase`, `recordResult`), matching the pattern already established for the memory capture path. State this explicitly in Phase 1's Architecture/Implementation Steps, not left to be inferred at implementation time.

## Finding 2: Single global `active-run.json` + Gate 1 slug-binding breaks concurrent same-directory sessions that the repo's own rules explicitly permit

- **Severity:** Critical
- **Location:** phase-01-state-evidence-loop-core.md Architecture (line 39, single file path, no session partition beyond an internal `session_id` field); phase-02-gate-hook-lifecycle-integrity.md Implementation Step 2 ("when `active-run.json` exists, require the newest plan and slug match with active-run"); phase-07-end-to-end-validation-matrix.md scenario 7 ("Two parallel runs on one project → conflicting writes detected/reconciled per authority rules (Markdown canonical)")
- **Flaw:** `active-run.json` is a single path (`session-state/active-run.json`) scoped by CWD/`CLAUDE_PROJECT_DIR`, not partitioned per session. Worktree isolation is only mandated for COMPLEX tasks (`parallel-execution-rules.md` Rule 6: "Only COMPLEX Tasks Qualify... TRIVIAL and STANDARD tasks use the default sequential pipeline"), meaning two ordinary STANDARD-tier sessions (e.g. two Claude Code windows open on the same repo, a common real workflow) both write into the same `session-state/` directory with no worktree separation. When session B activates its own run, it overwrites the global `active-run.json` pointer that session A's approved plan had set. Phase 2 then makes Gate 1 REQUIRE the active-run slug to match the plan being edited — so session A's legitimate, already-approved plan now fails Gate 1 because the global pointer was silently reassigned to session B's run. "Markdown canonical" (the stated authority rule for JSON-vs-Markdown content conflicts) does not resolve this: both sessions have valid, non-stale Markdown plans; the bug is about WHICH plan the shared pointer currently names, not about content correctness.
- **Failure scenario:** Dev has plan A approved and is mid-edit in window 1 (Gate 1 previously passed because active-run pointed at A). Dev opens window 2, runs `/mk:fix` on an unrelated bug (STANDARD tier, no worktree required), which activates run B and overwrites `active-run.json`. Window 1's next Edit call now fails Gate 1 with a "wrong-slug plan" deny — a hard stop on legitimate, previously-authorized work, with no in-plan mechanism to detect or reconcile the collision.
- **Evidence:** `meowkit/.claude/rules/parallel-execution-rules.md` Rule 6 (only COMPLEX requires worktrees); Phase 1 Architecture block only lists one file path for `active-run.json` with `session_id` as a payload field, not a partition key; Phase 2 line 40 literal requirement ("require the newest plan *and* slug match with active-run"); no per-session file naming (`active-run-{session_id}.json`) or advisory-lock arbitration is proposed anywhere in Phases 1–2, and Phase 7 scenario 7 only promises detection/reconciliation "per authority rules (Markdown canonical)" which does not cover pointer-identity collisions.
- **Suggested fix:** Either (a) scope the active-run file path by `session_id` and have Gate 1 resolve "the active run for THIS session" rather than "the global active run," or (b) explicitly document in Phase 1/2 that concurrent same-directory sessions are out of scope and must use worktree isolation regardless of task tier — a change to `parallel-execution-rules.md` Rule 6 the plan does not currently propose.

## Finding 3: Hooks gain a hard runtime dependency on the mewkit CLI/dist build, breaking today's zero-dependency hook design

- **Severity:** High
- **Location:** phase-01-state-evidence-loop-core.md Architecture (lines 32-38: "CLI: `state` subcommand... thin .cjs wrapper for hooks"); Related Code Files (line 52: "thin wrapper `.claude/hooks/lib/session-state.cjs`")
- **Flaw:** Every existing hook and `.cjs` handler in `.claude/hooks/` is self-contained — no hook currently requires, imports, or shells out to `packages/mewkit/src` or its compiled `dist/`. The plan's "thin .cjs wrapper" is described as wrapping the new `core/session-state/` module, which lives in TypeScript source (`packages/mewkit/src/core/session-state/`) — not directly `require()`-able by a `.cjs` hook without either (a) shelling out to `npx mewkit state ...` (requiring a resolvable/built package) or (b) duplicating writer logic in the wrapper itself (defeating "single typed writer API"). Neither path is specified. This introduces the exact class of fragility the toolkit's own hook design has avoided so far, and it lands on hooks Phase 2 turns into a hard, no-bypass gate (`gate-rules.md`: "These are hard stops. No automation may bypass them").
- **Failure scenario:** A downstream project consumes `.claude/` as a flat-copied install (per `docs-reference-contract.md`'s "shipped into other projects by the kit's installer" model) without `packages/mewkit` present locally, or `npx mewkit` cannot resolve (offline, registry outage, version pin mismatch). Every skill-driven `state` CLI call now fails. Depending on how the `.cjs` wrapper handles that failure, either (a) it fails open, silently producing no state writes — regressing to today's dead-loop behavior camouflaged as "fixed," or (b) it fails closed and the Phase 2 hard gates start blocking ALL work in every session, project-wide, until the CLI is available — neither of which is addressed in either phase's Risk Assessment.
- **Evidence:** `grep -rln "npx mewkit\|node.*dist/index" .claude/hooks/` returns only an instructional echo in `project-context-loader.sh:39` (never auto-invoked); no existing `.cjs` handler imports from `packages/mewkit` (`grep -rn "require(" .claude/hooks/handlers/*.cjs` shows only `checkpoint-utils.cjs` and local `fs`/`path`/`child_process`); `packages/mewkit/dist/` is gitignored (`.gitignore:5`) and only populated by a build step.
- **Suggested fix:** Phase 1 must explicitly specify the wrapper's dependency-resolution strategy (built-dist path resolution vs. bundled/precompiled reader, with a defined fail-open/fail-closed default) and add a doctor probe (Phase 2 already extends `doctor --hard-gates`) that verifies the state CLI is reachable BEFORE hooks depend on it for blocking decisions.

## Finding 4: No specified behavior for gate-enforcement.sh when `active-run.json` is corrupt/unparseable — ambiguous between fail-open (defeats the gate) and fail-closed (blocks all edits repo-wide)

- **Severity:** High
- **Location:** phase-02-gate-hook-lifecycle-integrity.md Implementation Step 2 (lines 40); Success Criteria (line 49: "Stale/unapproved/wrong-slug plan cannot unlock source writes while a different run is active")
- **Flaw:** The plan specifies matching semantics for stale/wrong-slug/absent active-run, but never states what `gate-enforcement.sh` does when `active-run.json` EXISTS but fails to parse (truncated by a crashed writer between steps, hand-edited, or written under a schema version the reader doesn't tolerate). `pre-completion-check.sh`'s existing Python parsing already demonstrates the toolkit is aware this needs handling — every JSON read there is wrapped in `try/except` with an explicit safe default (`print('1')` on exception, `print('')` on exception) — but Phase 2's plan for the NEW, stricter, no-bypass Gate 1 binding doesn't describe an equivalent contract.
- **Failure scenario:** A prior session was killed mid-write and left a `.tmp` file that never got renamed (rename didn't complete, so the OLD file persists — safe) OR a user/agent hand-edits `active-run.json` and introduces a syntax error while debugging. `gate-enforcement.sh` (a bash script, unlike the Python-driven pre-completion-check.sh) must invoke some JSON parse; if that parse throws and the surrounding bash doesn't have an explicit safe-default capture (`VAR=$(python3 -c ... 2>/dev/null)` silently yields empty string), the gate's behavior depends entirely on how "empty" is interpreted downstream — a single ambiguity in a component explicitly documented as "No automation may bypass" (`gate-rules.md`).
- **Evidence:** `pre-completion-check.sh:56-63` shows the existing defensive pattern (`try/except` → default `'1'`) that Phase 2 needs to replicate for the new binding but doesn't name; `gate-enforcement.sh:112-120` (Gate 1's existing glob-based check) has no JSON parsing today, so there's no established bash-side pattern to point to for the new active-run consumption.
- **Suggested fix:** Phase 2 Implementation Step 2 must explicitly define parse-failure handling for `active-run.json` (e.g., "malformed active-run.json degrades to the no-active-run existence-check fallback, with a warning" — consistent with the plan's existing "no-active-run fallback" risk mitigation on line 58) and add a test case for it (malformed active-run fixture) alongside the already-listed stale/wrong-slug/absent cases in Step 1's test list.

## Finding 5: Attempts-cap counter stays slug-keyed while evidence scoping moves to run_id — no rekey/migration path specified

- **Severity:** Medium
- **Location:** phase-02-gate-hook-lifecycle-integrity.md Implementation Step 3 ("Pre-completion: require `verification-required.json` scope fields to match active-run (`run_id`, `plan_slug`)... Keep the attempts cap.")
- **Flaw:** The current attempts cap in `pre-completion-check.sh` is a `{slug: count}` map keyed purely by plan slug (`ATTEMPTS_FILE`, keyed by `$SLUG`), with no run identity. Phase 2 rescopes verification evidence to `(run_id, plan_slug)` pairs but the plan only says "Keep the attempts cap" — it does not specify whether the cap becomes `(run_id, slug)`-keyed or stays slug-only. If it stays slug-only, two different `run_id`s against the same plan slug (e.g., a run abandoned and restarted, or resumed after a schema/session reset) share one counter — a fresh run inherits a stale run's near-exhausted attempt count, or a first-time run silently benefits from a previous run's already-consumed budget, defeating the "avoid infinite loop" rationale attached to the cap.
- **Failure scenario:** Run 1 on plan `foo-feature` exhausts 3 attempts (soft-nudges through, per current design at `pre-completion-check.sh:198-206`, which explicitly RESETS the counter to allow the next session a fresh start). Under Phase 2 scoping, if a NEW run_id is activated on the same slug in the same or a resumed session before the counter reset semantics are re-verified, behavior is undefined by the plan text — the reader cannot tell from the phase file whether the intended keying is `slug` or `run_id+slug`.
- **Evidence:** `pre-completion-check.sh:96-105` (attempts read, `d.get('$SLUG', 0)`), `:113-124` (attempts write, `d['$SLUG'] = ...`) — confirmed slug-only keying today; Phase 2 text has no explicit statement of the new key shape.
- **Suggested fix:** State explicitly in Phase 2 Step 3 whether the attempts file becomes `(run_id, slug)`-keyed, and if so, define migration behavior for any slug-only entries already on disk from pre-Phase-2 sessions.

## Finding 6: RESULT envelopes accumulate inside the git-tracked `tasks/plans/` tree with no retention/gitignore policy

- **Severity:** Medium
- **Location:** phase-01-state-evidence-loop-core.md Architecture (line 42: `tasks/plans/<plan>/results/RESULT-<run_id>.json`); Implementation Step 4
- **Flaw:** `tasks/plans/` is not gitignored (confirmed — no `results/` or `tasks/plans/*/results` entry in `.gitignore`, and plan.md/phase files are the toolkit's canonical, version-controlled planning artifacts). `recordResult` is designed to write a new envelope file on every recorded run (build, verify, review iterations can each produce a `RESULT-<run_id>.json`), and nothing in Phase 1 specifies whether these are gitignored, pruned, or squashed. This directly risks repo bloat for any plan that iterates multiple times (which the harness's own iteration cap of "3 rounds" and re-run-on-fail patterns make routine).
- **Failure scenario:** A plan goes through 3 build/verify/review iteration rounds (per `harness-rules.md` Rule 4's own 3-round cap) plus resumed sessions after kills — each producing a fresh `RESULT-<run_id>.json` under a tracked directory. Over the life of an active project with dozens of plans, this is dozens-to-hundreds of small JSON files committed alongside the Markdown plans they're meant to support, with no cleanup story, contradicting the plan's own "no context garbage" global constraint at the repository-hygiene level (distinct from, but analogous to, the context-injection concern the constraint states explicitly).
- **Evidence:** `.gitignore` has no `results/` or `tasks/plans/*/results` line (checked); `tasks/plans` exists as a real, non-ignored directory today (`find tasks -maxdepth 3 -type d` confirms `tasks/plans` is present and not excluded).
- **Suggested fix:** Either gitignore `tasks/plans/*/results/` (results are derived state, consistent with the plan's own "Authority order: JSON is compact derived state") and rely on `last_result_ref` pointers for what matters, or define an explicit pruning step (age-based, mirroring `mk:memory --prune`'s >90-day pattern already in the codebase).

## Finding 7: Mid-rollout asymmetry — Phase 1 makes the Stop-side gate live and blocking before Gate 1's write-side scoping (Phase 2) lands, with no risk called out

- **Severity:** Medium
- **Location:** phase-01-state-evidence-loop-core.md Implementation Step 6 ("Wire producers: plan activation..., phase transition + verification-required..., result recording..."); Risk Assessment (lines 78-82, no mention of this asymmetry); phase-02-gate-hook-lifecycle-integrity.md (Gate 1 binding is Phase 2 only)
- **Flaw:** Today, `verification-required.json` has zero producers, so `pre-completion-check.sh` always exits 0 (`pre-completion-check.sh:50-53`, confirmed) — Stop is never blocked. Phase 1 Step 6 wires REAL producers into `cook`/`autobuild`/`fix`/`review` skill prose so `verification-required.json` starts getting written in production for the first time. This makes the Stop gate genuinely block sessions for the first time — while Gate 1 (the entry-side plan-approval check) remains completely untouched until Phase 2 and still accepts "any `tasks/plans/*/plan.md` exists" (`gate-enforcement.sh:112-120`, confirmed unscoped). Between shipping Phase 1 and shipping Phase 2 — a gap the plan's own dependency graph allows to be arbitrarily long, since Phase 2 merely "depends on 1" rather than being bundled with it — users get a system where the exit gate is newly strict and can genuinely trap a session, while the entry gate remains maximally permissive (any stale/unrelated plan unlocks writes). This inverts the intended enforcement order (loose entry, strict exit) without the plan flagging it as a rollout risk anywhere.
- **Failure scenario:** Phase 1 ships. A developer starts unrelated exploratory work with a stale, unrelated `plan.md` sitting in `tasks/plans/` (passes Gate 1 trivially, as today). The now-live producer wiring causes `cook`/`review` to write `verification-required.json` for the actual active work. Session Stop now blocks (up to 3 attempts) because the evidence check (`pre-completion-check.sh`'s existing HAS_VERIFICATION logic) can't find a matching verdict/contract/trace for a slug that was never really gated at entry — the user experiences a brand-new hard stop with no corresponding tightening of the write-side gate that would have prevented the underlying problem (unscoped plan) in the first place.
- **Evidence:** `pre-completion-check.sh:50-53` (today's silent no-op, confirmed); Phase 1 Related Code Files (line 53) explicitly lists skill instruction changes for producer wiring but not `gate-enforcement.sh`; Phase 2 Related Code Files (line 33) is the first phase to touch `gate-enforcement.sh`.
- **Suggested fix:** Either explicitly sequence Phase 1 and Phase 2 as a single ship unit (not two independently-mergeable phases), or add a Phase 1 risk-assessment line acknowledging the asymmetric window and specifying a mitigation (e.g., keep Stop-side enforcement behind a flag until Phase 2's Gate 1 scoping also lands).

## Unresolved Questions

- Does the plan intend `session-state/` to ever be session-scoped (e.g., filename includes `session_id`), or is the single-global-file design (Finding 2) deliberate and parallel-session collisions accepted as out of scope? The plan text doesn't say either way.
- For Finding 3 (CLI dependency), is the intended distribution model "hooks always assume a locally-built `packages/mewkit`" (dev-only, i.e., this toolkit's own dogfooding never ships this way to end users) or "hooks assume `npx mewkit` resolves the published package" (network-dependent for every gate/verification call)? This materially changes the severity and the fix.
