# Red-Team Review — Harness Execution-Loop Hardening Plan

Scope: plan.md + 7 phase files, verified against `meowkit/` source (`.claude/hooks/`, `packages/mewkit/src/`). No lint/build/test run — plan-document review only.

## Finding 1: Phase 1's enforcement backstop claim is false for 3 of 4 new state fields — the audit's core failure mode is relocated, not solved

- **Severity:** Critical
- **Location:** phase-01-state-evidence-loop-core.md Risk Assessment ("Producers are prose-executed by the model... mitigate by making Phase 2 gates *consume* this state, so skipping producers visibly blocks progress instead of silently passing"); phase-02-gate-hook-lifecycle-integrity.md Implementation Steps 2–3.
- **Flaw:** `active-run.json`'s schema (phase-01:39-41) has 8 fields: `run_id, session_id, plan_path, plan_slug, phase, blockers[], open_decisions[], last_result_ref`. Phase 2's only two consumers of this file — Gate 1 (step 2) and pre-completion (step 3) — read exactly `run_id`/`plan_slug` (existence + match). Neither step reads, validates, or gates on `phase`, `blockers`, or `open_decisions`. So the claimed mitigation ("skipping producers visibly blocks progress") is true only for the `activateRun` call (Gate 1 needs *a* run to exist with matching slug); `setPhase`, `addBlocker`, `recordDecision` calls remain exactly as skippable as today's aspirational `HOOKS_INDEX.md` producer claims (phase-01 Key Insight: "grep of those skills finds zero write instructions. Docs are aspirational") — just one field deeper.
- **Failure scenario:** A skill calls `state activate-run` once at plan-creator's final step (satisfying Gate 1), then never calls `state set-phase`/`add-blocker` during cook/autobuild's actual phase transitions (prose CLI calls are exactly what the audit found gets skipped under time pressure or context compaction). `active-run.phase` stays at its initial value indefinitely. Gate 1 keeps passing (slug still matches). The kill/resume drill (Success Criteria: "restores plan, phase, blockers, next action") silently restores stale/wrong data with no gate ever detecting it — the resume experience regresses from "empty file, obviously broken" to "populated file, plausibly wrong," which is harder to catch.
- **Evidence:** `phase-01-state-evidence-loop-core.md:39-41` (schema), `phase-02-gate-hook-lifecycle-integrity.md:40-41` (Gate 1/pre-completion only check `run_id`/`plan_slug`), `.claude/hooks/gate-enforcement.sh:105-130` (current Gate 1 logic — plan existence/slug only, confirms no phase-aware check exists to extend).
- **Suggested fix:** Either (a) state explicitly in Phase 1/2 that `phase`/`blockers`/`open_decisions` are display-only/advisory (matching reality, per the plan's own "Claims cleanup" global constraint), or (b) add a Phase 2 consumer that validates `phase` monotonically advances and flags staleness (e.g., `active-run.updated_at` older than N tool calls since last phase change) so the claim in the Risk Assessment is actually true.

## Finding 2: Phase 1 proposes a new atomic-write JSON state module without acknowledging the existing one it duplicates

- **Severity:** High
- **Location:** phase-01-state-evidence-loop-core.md Architecture diagram and Related Code Files ("Create: `packages/mewkit/src/core/session-state/`... thin `.cjs` wrapper `.claude/hooks/lib/session-state.cjs`"); Non-functional requirement "atomic writes (tmp+rename)".
- **Flaw:** `.claude/hooks/lib/shared-state.cjs` already implements exactly this primitive — generic `load(name)`/`save(name, data)` over `session-state/*.json` with tmp+rename atomic writes — and is already injected into every hook handler via `dispatch.cjs` (used today by `checkpoint-writer.cjs` as `state.load('active-plan')`, `state.load('detected-model')`, etc.). The plan's proposed `.claude/hooks/lib/session-state.cjs` is a near-identical name with overlapping responsibility, created without any reference to `shared-state.cjs` in either the Key Insights or Related Code Files sections.
- **Failure scenario:** Two atomic-write JSON helpers ship side by side (`shared-state.cjs` used by existing handlers like `checkpoint-writer.cjs`, `orientation-ritual.cjs`; new `session-state.cjs` used by the new `state` CLI wrapper). A future contributor fixes a write-race or schema bug in one and never touches the other, because nothing in the plan documents that they're siblings. This is exactly the "parallel reimplementation of existing utilities" AI-risk pattern.
- **Evidence:** `.claude/hooks/lib/shared-state.cjs:10-26` (load/save, tmp+rename); `.claude/hooks/dispatch.cjs` (only consumer of `shared-state.cjs`, confirmed via grep); `.claude/hooks/handlers/checkpoint-writer.cjs:16-18` (`state.load(...)` calls sourced from `shared-state.cjs` via dispatch injection).
- **Suggested fix:** Phase 1 Implementation Step 2/3 should explicitly build the new session-state module ON TOP OF `shared-state.cjs` for the `.cjs`-hook side (or state a deliberate reason for a second primitive — e.g., schema versioning that `shared-state.cjs` lacks — and note it in Key Insights so the duplication is a documented trade-off, not an oversight).

## Finding 3: Phase 2's Gate 2 verdict-parsing plan ignores the `verdict-gate` CLI that already does this, more rigorously

- **Severity:** High
- **Location:** phase-02-gate-hook-lifecycle-integrity.md Implementation Step 3 ("parse the newest `tasks/reviews/*-verdict.md` decision — empty, missing, or negative decision blocks") and Step 4 (Gate 2 hook-enforcement). Plan mentions `verdict-gate` only twice, both about CLI-help omission (phase-04:22,53) and e2e coverage (phase-07:38) — never as reusable Gate 2 machinery.
- **Flaw:** `packages/mewkit/src/commands/verdict-gate.ts` + `packages/mewkit/src/memory/verdict-schema.ts` already implement a Zod-validated `VerdictSchema` (decision: PASS/PASS_WITH_RISK/BLOCKED), `evaluateVerdictGate()` that resolves slug→JSON path, parses, and returns typed pass/blocked/invalid/missing/legacy-md status with exit codes 0/1/2 — already invoked by `mk:review` step-04 as a self-check (`.claude/skills/review/step-04-verdict.md:136`, "run `npx mewkit verdict-gate <slug>`"). Phase 2 Step 3 instead describes writing fresh logic to "parse the newest `tasks/reviews/*-verdict.md` decision" against the **Markdown** file directly, not the JSON companion the existing gate already validates — a different, less rigorous parsing target than what already exists and already ships to the review skill.
- **Failure scenario:** Two independent verdict-parsing implementations ship: `evaluateVerdictGate()` (JSON-schema-validated, used by `mk:review`'s self-check) and a new hook-side Markdown-decision parser (Phase 2, used by `pre-completion-check.sh`/Gate 2). They can disagree — e.g., a verdict JSON says `BLOCKED` but the accompanying `.md` decision line is stale/hand-edited to look affirmative, and the hook-side parser (reading only `.md`) passes while `verdict-gate` would have blocked. The "verdict format variance breaks parsing" risk the plan already flags (Risk Assessment, phase-02) is self-inflicted by not reusing the schema that already prevents that variance.
- **Evidence:** `packages/mewkit/src/commands/verdict-gate.ts:1-30`; `packages/mewkit/src/memory/verdict-schema.ts:16-27,44-82`; `.claude/skills/review/step-04-verdict.md:136`.
- **Suggested fix:** Phase 2 Step 3/4 should invoke `evaluateVerdictGate()` (or shell out to `mewkit verdict-gate <slug>`, exit-code driven) from `pre-completion-check.sh`/the Stop path, not re-parse the `.md` file independently. Phase 4's CLI-help fix for `verdict-gate` should move earlier or be folded into Phase 2 since Phase 2 becomes its primary caller.

## Finding 4: "Active plan" resolution is already fragmented across 3 mechanisms; Phase 1 fixes 1 of them, leaving the orchviz UI to disagree with gates by construction

- **Severity:** High
- **Location:** phase-01-state-evidence-loop-core.md Related Code Files (only lists `packages/mewkit/src/orchviz/overlay/session-state-readers.ts` as the orchviz touch point).
- **Flaw:** Grep confirms three pre-existing, independent "what is the active plan" resolvers: (1) `session-state/active-plan.json` (`{path}`, read by `pre-completion-check.sh` and `checkpoint-writer.cjs`), (2) legacy plaintext `session-state/active-plan` fallback (read by `pre-completion-check.sh`, explicitly forbidden as a resolution source by `plan-creator/step-08b-html-render.md:37` per phase-01's own Key Insights), and (3) `findActivePlan()` in `packages/mewkit/src/orchviz/plan/find-active-plan.ts` — a completely separate mtime-based "most recently touched `tasks/plans/*/plan.md`" heuristic that drives the orchviz dashboard's plan-switcher, plan-table, and top-strip UI (confirmed consumers: `plan-switcher/*.tsx`, `plan-table/*.tsx`, `top-strip/index.tsx`). Phase 1 adds a 4th (`active-run.json`) and only touches the `session-state-readers.ts` "overlay" reader — it does not touch `find-active-plan.ts` or its UI consumers.
- **Failure scenario:** After Phase 1, `active-run.json` becomes the gate-enforced canonical "active run," but the orchviz dashboard (the human-facing surface for resuming work, per the plan's own kill/resume drill intent) still shows whatever plan directory has the newest mtime — which can be a plan a human is casually browsing/editing, not the one gates are enforcing. A user watching the dashboard sees plan A as "active" while Gate 1 is actually bound to plan B's slug. This directly undermines the plan-level acceptance criterion "Kill/resume drill restores plan, phase, blockers, next action" — the drill can pass against `active-run.json` while the UI users actually look at shows something else.
- **Evidence:** `packages/mewkit/src/orchviz/plan/find-active-plan.ts:21-30` (independent mtime-based resolution, scoped to `tasks/plans/`, no session-state read); grep of `active-plan` across `packages/mewkit/src/orchviz-web/components/{plan-switcher,plan-table,top-strip}/*` confirms UI dependency on this resolver, not on `session-state-readers.ts`.
- **Suggested fix:** Phase 1 scope (or an explicit Phase 1 follow-up note) should either point `findActivePlan()` at `active-run.json` when present (falling back to mtime heuristic only when no run is active), or explicitly document in Requirements that orchviz's UI "active plan" and the gate-enforced "active run" are intentionally different concepts with different audiences — the plan currently does neither.

## Finding 5: Phase 4 declares a dependency on Phase 1 only, but its Implementation Step 2 requires an artifact Phase 2 produces — breaks the stated "4 and 5 can run in parallel"

- **Severity:** Medium
- **Location:** phase-04-inventory-as-capability-authority.md frontmatter (`dependencies: [1]`) and Implementation Step 2 ("source `producer`/`consumer`/`enforcement` for hooks from the Phase 2 hook manifest instead of hand-typing"); plan.md Dependencies section ("Phases 4–5 depend on 1 ... 4 and 5 can run in parallel").
- **Flaw:** The hook manifest (`hook-manifest.json`, with "producer/consumer/enforcement level") is a Phase 2 deliverable (phase-02-gate-hook-lifecycle-integrity.md Implementation Step 5: "Build the hook-manifest generator"). Phase 4 explicitly consumes that exact artifact for its own hook-related inventory fields. This is an undeclared cross-phase dependency: Phase 4 cannot correctly populate hook `producer`/`consumer`/`enforcement` until Phase 2 ships, contradicting the plan-level claim that Phase 4 depends only on Phase 1 and can run parallel to Phase 5.
- **Failure scenario:** A team starts Phase 4 in parallel with Phase 2 (as the plan's own Dependencies section licenses), reaches Step 2, and either hand-types the hook fields anyway (reproducing the exact "count drift" / hand-maintained-data problem Phase 4 exists to eliminate — phase-04 Key Insights: "Count drift is structural: README/index counts are hand-typed and re-drift") or blocks mid-phase waiting on Phase 2, invalidating the parallelization the plan promised.
- **Evidence:** `phase-04-inventory-as-capability-authority.md:7` (`dependencies: [1]`), `:39` (Step 2 references "the Phase 2 hook manifest"); `phase-02-gate-hook-lifecycle-integrity.md:43` (Step 5, manifest generator is a Phase 2 output); `plan.md:47` ("4 and 5 can run in parallel").
- **Suggested fix:** Add Phase 2 to Phase 4's `dependencies` frontmatter and update plan.md's Dependencies section, OR resequence Phase 4 Step 2 to hand-populate hook fields temporarily and backfill from the manifest once Phase 2 lands (explicitly noted as a two-pass step).

## Finding 6: "Markdown wins" conflict-detection has no defined algorithm — no parser exists to map plan.md/phase-file checkbox state to a comparable `phase` value

- **Severity:** High
- **Location:** phase-01-state-evidence-loop-core.md Implementation Step 1 ("conflict rule (active-run pointing at plan whose Markdown checkboxes disagree → Markdown wins and state is marked stale)"); plan.md global constraint ("Authority order: Markdown plans/checkboxes stay canonical; JSON is compact derived state; SQLite only ever a derived index. State conflicts always lose to canonical Markdown").
- **Flaw:** No existing code parses plan.md's phase status table or phase-file "Todo List" checkboxes into a machine-comparable phase value — grep across `packages/mewkit/src/commands/task.ts` (the closest candidate, confirmed by Phase 5's own Key Insight: "`task.ts` only does `tasks/` file CRUD") and the templates directory finds zero checkbox-parsing logic. Phase files use free-form `- [ ] item` todo lists (per `documentation-management.md`'s Phase File Structure), not a single canonical "current phase" marker — a plan can have Phase 2 fully checked, Phase 3 half-checked, and Phase 4 untouched simultaneously, which has no unambiguous mapping to a single `active-run.phase` integer. The plan's own Success Criteria ("state conflict resolves in Markdown's favor") assumes this mapping is well-defined; it is not specified anywhere in Phase 1.
- **Failure scenario:** `activeRun.phase = 3` (JSON says Build), but plan.md's phase table shows Phase 2 (Test) still "In Progress" and Phase 4 (Review) has some pre-checked boxes from a copy-pasted template. The conflict detector has no deterministic rule to decide "what does Markdown actually say the phase is" — it either guesses (defeating "Markdown wins" as a real authority rule) or the feature ships as an unimplemented no-op behind a Success Criteria checkbox nobody can verify.
- **Evidence:** `documentation-management.md` (Phase File Structure — todo list is unstructured checkboxes, no canonical single-phase marker); grep of `packages/mewkit/src/commands/task.ts` and `tasks/templates/` for checkbox-state parsing returns nothing; `phase-01-state-evidence-loop-core.md:58` (states the conflict rule as a one-line test case without defining the parsing contract).
- **Suggested fix:** Define the canonical phase-detection rule explicitly in Phase 1 (e.g., "phase = highest-numbered phase file whose frontmatter `status: done` AND its Todo List has zero unchecked boxes, plus one" or similar) before claiming it as tested/enforced behavior, or scope Phase 1's conflict rule down to plan-slug/existence only (already covered by Gate 1) and drop the phase-level Markdown-authority claim until a parser exists.

## Finding 7: Phase 2's "unscoped evidence" diagnosis overstates the actual gap — 3 of 4 pre-completion checks are already slug-scoped

- **Severity:** Medium
- **Location:** phase-02-gate-hook-lifecycle-integrity.md Key Insights ("Pre-completion... evidence can be unscoped (any file existence / any trace pass). Verdict content is not parsed.") and Requirements ("pre-completion requires scoped evidence and a parsed, affirmative verdict").
- **Flaw:** Reading `pre-completion-check.sh` in full shows it already resolves `ACTIVE_PLAN_SLUG` (from `active-plan.json`/`verification-required.json`) and glob-matches 3 of its 4 evidence checks against that slug: `tasks/reviews/[0-9]*-"$SLUG"-evalverdict.md`, `tasks/contracts/*-"$SLUG"-sprint-*.md`, and `tasks/reviews/[0-9]*-"$SLUG"-verdict.md`. Only evidence check #3 (the trace-log scan: `tail -n 500 .claude/memory/trace-log.jsonl | grep -qE '"test_result":"pass"|...'`) is genuinely unscoped — it matches ANY recent pass in the trace log regardless of slug. The plan's Key Insight describes the whole gate as unscoped ("any file existence... any trace pass"), which is only true for 1 of 4 paths.
- **Failure scenario:** Not a runtime failure — a plan-accuracy risk. Because the diagnosis is broader than the actual gap, Phase 2 Step 3 may rebuild slug-scoping that already exists (wasted effort) instead of targeting the one real hole (the trace-log check), and reviewers approving Phase 2 based on the stated Key Insight will underestimate how much of Gate 2/pre-completion is already correctly scoped.
- **Evidence:** `.claude/hooks/pre-completion-check.sh` lines resolving `ACTIVE_PLAN_SLUG` and the 4 `HAS_VERIFICATION` checks (evalverdict/contract/trace-log/verdict, in that order) — checks 1, 2, 4 use `"$SLUG"` in their glob; check 3 (trace-log) does not reference `$SLUG` at all.
- **Suggested fix:** Correct the Key Insight to name the trace-log check specifically as the unscoped path, and scope Phase 2 Step 3 to fixing that one check (plus verdict-content parsing per Finding 3) rather than implying a full rebuild of pre-completion's evidence scoping.
