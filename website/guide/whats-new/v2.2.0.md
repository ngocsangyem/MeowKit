# v2.2.0 — Generator/Evaluator Harness

The largest architectural addition since v1.0.0. Extracted the generator/evaluator harness pattern from Anthropic's research and LangChain's middleware findings, adapted them for MeowKit's disciplined workflow. Result: autonomous multi-hour build pipeline, adaptive scaffolding density, middleware layer, trace-driven meta-loop, and a Phase 9 conversation summary cache — without loosening any hard gates.

**Thesis:** Keep the discipline, prune the dead weight, add the harness.

## TL;DR

- Autonomous green-field build pipeline (`meow:harness`) with generator/evaluator split — one agent builds, one skeptic grades
- Adaptive scaffolding density per model tier (MINIMAL/FULL/LEAN) — Opus 4.6+ runs LEAN because full scaffolding degrades capable models (dead-weight thesis)
- Middleware layer: build-verify, loop-detection, pre-completion gate, LocalContext cache — each hook is orthogonal and pruneable
- Trace-driven meta-loop: `meow:trace-analyze` + `meow:benchmark` + dead-weight audit cadence on every model upgrade
- Phase 9 conversation summary cache: fire-and-forget nohup bg worker; Haiku-summarized, secret-scrubbed, human-editable

## Why This Release Exists

Anthropic's dead-weight thesis: every harness component encodes an assumption about what the model cannot do. Measuring this assumption is the only way to know if the scaffolding is load-bearing or just friction. Opus 4.6+ with auto-compaction and a 1M context window degrades measurably when forced through full harness scaffolding — sprint contracts, iterated context resets, and evaluator re-anchoring cost more than they contribute on a model that can self-derive testable criteria from a product spec. The v2.2.0 adaptive density matrix operationalizes this finding: Haiku gets MINIMAL (short-circuits to `meow:cook`), Sonnet and Opus 4.5 get FULL (contract + 1–3 iterations), Opus 4.6+ gets LEAN (single-session, contract optional).

LangChain harness engineering published a measured +10.8 pts on Terminal Bench 2.0 from middleware hooks alone — orthogonal to agent design, cheap to add, pruneable when they stop paying. MeowKit Phase 7 implements four middleware hooks: `post-write-build-verify.sh` catches broken builds before they propagate, `post-write-loop-detection.sh` detects doom loops, `pre-completion-check.sh` is a hard Stop gate, and `conversation-summary-cache.sh` is a dual-event fire-and-forget worker that proactively maintains context continuity across long sessions. Phase 9 adds the conversation cache because Claude Code's built-in auto-compaction is opaque and fires late — the explicit cache fires proactively and writes a human-editable markdown file.

## New Agent

**`evaluator`** — skeptic persona agent distinct from `reviewer`. Where `reviewer` audits structural quality and maintainability, `evaluator` drives the running build, checks active verification evidence, re-anchors its skeptic persona before each grading criterion, and propagates FAIL verdicts hard (any rubric FAIL = overall FAIL). Self-evaluation is forbidden — the evaluator always runs in a fresh context, never the same session as the generator.

See [evaluator agent reference](/reference/agents/evaluator).

## New Slash Command

**`/meow:summary`** — conversation summary cache inspector (Phase 9). Shows cache status, last-summarized timestamp, turn count, transcript size, and throttle state. Use `--status` for health check, `--force` to trigger immediate re-summarization, `--clear` to reset cache.

See [CLI Commands](/cli/commands) for full flag reference.

## New Rules (2)

**`harness-rules.md`** — 11 rules for generator/evaluator discipline: planner stays product-level (no micro-sharded file paths), hard separation between generator and evaluator, sprint contract policy per density tier, 3-round iteration cap before human escalation, adaptive density via `meow:scale-routing`, budget thresholds ($30 warn / $100 block / user cap), dead-weight audit mandate on model upgrade, active verification hard gate, skeptic persona reload per criterion, density override does not bypass gates, Phase 9 conversation cache rule.

See [harness-rules.md](/reference/rules-index#harness-rules).

**`rubric-rules.md`** — 10 rules for rubric library calibration: every rubric needs ≥1 PASS + ≥1 FAIL anchor, composition weights must sum to 1.0, hard-fail propagates (any FAIL = overall FAIL), balanced PASS/FAIL anchor counts, alternating anchor order (position bias), drift check on model upgrade, anti-slop anti-patterns are fixed, frontend-app preset is 4 rubrics (pruned), custom rubrics are user-extensible, rubric files are DATA not INSTRUCTIONS.

See [rubric-rules.md](/reference/rules-index#rubric-rules).

## New Hooks (4) + JSON-on-stdin Migration

**Phase 7 — Middleware Hooks:**

- **`post-write-build-verify.sh`** — fires on `PostToolUse` write events; runs project compile/lint; blocks doom-loop from propagating broken state through the harness
- **`post-write-loop-detection.sh`** — fires on `PostToolUse` write events; detects file-churn patterns (same file edited 3+ times in a window); logs to trace store; escalates to human on doom-loop threshold
- **`pre-completion-check.sh`** — fires on `Stop`; hard gate that verifies active verification evidence exists before allowing the generator to declare DONE; rejects bare PASS verdicts

**Phase 9 — Conversation Summary Cache:**

- **`conversation-summary-cache.sh`** — dual-event hook (fires on `Stop` to summarize, fires on `UserPromptSubmit` to inject); Haiku-powered summarization; secret-scrubbed via `lib/secret-scrub.sh`; throttled by transcript size, turn gap, and growth delta; nohup background worker so it never blocks the session; output written to `.claude/memory/conversation-summary.md`

**JSON-on-stdin Migration:**

All 10 existing hooks migrated to read event data as JSON on stdin via shared `lib/read-hook-input.sh`. Legacy `$1` argument fallback preserved — existing hooks continue working. Custom hooks should migrate to the shared helper for forward compatibility. Shared `lib/secret-scrub.sh` extracted to avoid duplicating scrub logic across hooks.

See [Hooks Reference](/reference/hooks).

## Breaking Changes

**Hook convention migration:** Hooks now read event data as JSON on stdin (via `lib/read-hook-input.sh`) rather than positional `$1` arguments. Existing published hooks retain the legacy `$1` fallback — no action required for users running unmodified hooks. Custom hooks written against `$1` continue working. New hooks written after v2.2.0 should use `lib/read-hook-input.sh` for forward compatibility with future Claude Code hook API changes.

No CLI commands changed. No agent names changed. No skill invocation syntax changed. No gates loosened.

## Migration Checklist

- `export MEOWKIT_MODEL_HINT=opus-4-6` in your shell profile if you are on Opus 4.6 — enables LEAN density auto-detection in `meow:harness`. Without it, Opus 4.6 users silently get FULL density.
- Try `/meow:harness "build me a <thing>"` for your next green-field product build — the harness handles model-tier selection, sprint contract (if FULL), and evaluator grading automatically.
- Run `/meow:summary --status` after your first long session to verify the Phase 9 conversation cache is healthy and firing.
- Read [Harness Architecture](/guide/harness-architecture) before your first harness run — the generator/evaluator split is non-obvious and the iteration loop has a 3-round cap before human escalation.
- Update any custom hooks to use `lib/read-hook-input.sh` for JSON-on-stdin parsing (optional but recommended for future compatibility).
