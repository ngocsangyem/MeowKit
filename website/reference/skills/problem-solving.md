# mk:problem-solving

Seven techniques for strategic stuck-ness. Each targets a specific failure mode of default thinking. NOT for debugging (use `mk:sequential-thinking`); NOT for solution exploration before planning (use `mk:brainstorming`); NOT for re-examining existing output (use `mk:elicit`).

## When to use

Match symptom to technique:

| Stuck symptom | Technique |
|---|---|
| Same thing 5+ ways, growing special cases | Simplification Cascades |
| Conventional solutions inadequate, need breakthrough | Collision-Zone Thinking |
| Same issue across domains, reinventing wheels | Meta-Pattern Recognition |
| Solution feels forced, "must be this way" | Inversion Exercise |
| "Should scale fine" — no idea where limits are | Scale Game |
| Told impossible / reasoning by analogy | First Principles |
| Bloated system, remove > add | Via Negativa |

## Process

1. Match symptom to ONE technique using the table above
2. Apply only that technique — do not stack multiple
3. If technique resolves the block → proceed
4. If not → combine 2 techniques (e.g., Scale + Via Negativa)
5. Explicit boundary: if "code broken" → reroute to `mk:sequential-thinking`
